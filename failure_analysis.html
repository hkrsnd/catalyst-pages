Case Study 3 Invalid SMILES

no_critic_20t_2r / chatdrug_004

The Medicinal Chemist proposed 0 molecule(s), but none could be parsed by RDKit into a valid molecular structure. Starting LogP was 6.14, with the optimization goal to reduce it.

With zero valid evaluations, the Project Lead correctly terminated:

“No viable proposals were generated in Round 1, and the current lead still has logP = 6.14 (well above the target). Without any improvement toward the primary objective, further rounds are unlikely to yield progress.”

This is a SMILES generation quality issue. The LLM generates character sequences that look like SMILES but contain subtle errors (unclosed rings, invalid valences, broken bracket atoms). The system currently has no retry mechanism — once proposals fail validation, the round is lost.

Root Cause Analysis

The underlying mechanisms behind each failure category

Root Cause 1: Project Lead Reading Comprehension

The Project Lead receives a generic instruction: “Review all evaluations and recommendations from this round.” It must read back through the full multi-agent conversation (7 messages with complex markdown tables) to find the evaluation data.

With long conversations and markdown-formatted property tables, gpt-oss-120b loses track of the actual numbers and hallucinates that data is missing. This is exacerbated when the Computational Chemist's text rendering shows None for similarity (even though the tool computed it).

Impact: 50% of all failures (4 out of 8) are caused by this issue. These are the most wasteful failures because valid, improved molecules exist but are discarded.

Root Cause 2: SMILES Generation Quality

The Medicinal Chemist generates SMILES strings that fail RDKit parsing. These contain subtle syntax errors: unclosed ring numbers, invalid valences, or broken stereochemistry annotations.

The system has no retry mechanism: once a proposal fails validation during extract_evaluations, it is recorded as invalid and the round proceeds. If all proposals are invalid, the Project Lead must terminate.

Impact: 50% of all failures (4 out of 8). Unlike Root Cause 1, these represent genuine LLM generation limitations, though a retry mechanism could recover from them.

Proposed Solutions

Actionable fixes ranked by expected impact

Highest Impact

Inject Structured Evaluation Summary into Decide Prompt

Instead of relying on the PL to read back through the CC's free-text narrative, inject a programmatically-formatted evaluation summary directly into the decide instruction.

How: After extract_evaluations(), format the ground-truth results (SMILES, LogP, delta, similarity) as a simple table and pass it as a template variable into the decide prompt.
Expected effect: Eliminates all 4 PL reading comprehension failures immediately. The PL sees clean, unambiguous data instead of having to parse markdown from prior messages.
Medium Impact

Retry on All-Invalid Proposals

When all proposals from the MC are invalid SMILES, instead of proceeding to the decide step (which inevitably terminates), retry the propose step with feedback about what went wrong.

How: After evaluation, if all(not ev.is_valid for ev in round_evals), re-run the propose step with an error message listing which SMILES failed and why.
Expected effect: Addresses the 4 invalid-SMILES failures by giving the MC a second chance. Many of these errors are minor syntax issues that the LLM can correct when told about them.
Safety Net

Decision Validation Guardrail

After the PL issues a TERMINATE decision, check if valid evaluations with objective improvements exist. If so, automatically advance the best candidate instead of terminating.

How: In lab_session.py, after _process_decision(), verify that termination is justified by checking round_evals for improvements.
Expected effect: A safety net that catches any remaining PL reading failures. Would recover all Category 1 failures even if the prompt fix is imperfect.
